# Teaming Guide

## Назначение

Этот файл описывает, как должны синхронизироваться фронтендер и бэкендер, чтобы:

- быстро собрать demo-first сценарий;
- не блокировать друг друга;
- после демо без хаоса перейти в MVP;
- строить экраны на реальных или хотя бы frozen контрактах.

## Главный принцип

Фронтенд не должен:

- ждать весь бэкенд целиком;
- выдумывать payload сам;
- строить “демо-экраны”, которые потом придется выбросить.

Бэкенд не должен:

- держать фронт без shape payload;
- менять ключи и статусы без handoff;
- считать, что “фронт потом подстроится”.

Рабочая схема такая:

1. Бэкенд идет немного впереди.
2. Бэкенд отдает frozen контракт.
3. Фронтенд сразу собирает экран под него.
4. Пока фронт собирает экран, бэкенд уже идет дальше.

---

## Как фронтенду пинговать бэкенд

Фронтенд пингует бэкенд, когда:

1. стартует новый экран;
2. не хватает поля в payload;
3. shape ответа вызывает костыль в UI;
4. надо понять, можно ли временно жить на mock.

Формат нормального пинга:

1. какой экран;
2. какой endpoint;
3. какого поля/статуса не хватает;
4. почему это ломает или усложняет UI.

Пример:

```md
Экран: /vibe/result
Endpoint: POST /api/v1/profile/voice
Проблема: не хватает отдельного короткого title для карточки вайб-паспорта
Почему важно: иначе фронт генерирует title сам и он расходится с продуктовой логикой
```

---

## Как бэкенду пинговать фронтенд

Бэкенд должен пинговать фронтенд, когда:

1. endpoint готов;
2. payload frozen;
3. есть пример успешного ответа;
4. понятно, mock там или real.

Формат:

```md
Endpoint: GET /api/v1/map/locations
Статус: mock-safe, shape frozen
Используется в: map screen
Пример ответа: ...
Ограничения: пока curated points, без реального bbox ranking
Можно брать во фронт: да
```

---

## Когда пинговаться обязательно

## 1. Перед стартом новой фазы

Нужно коротко сверить:

1. что именно делаем;
2. что считается минимальной готовностью;
3. что может заблокировать;
4. real это будет или временно mock-safe.

## 2. На handoff

Это главный рабочий пинг.

Без handoff фронт не должен считать контракт надежным.

## 3. Перед freeze на съемку

Нужно договориться:

1. какие экраны снимаются;
2. какие endpoint’ы frozen;
3. какие моки допустимы;
4. что запрещено менять до конца захвата.

## Что фронтенду можно делать без готового бэкенда

Фронт может идти вперед, если:

1. контракт уже frozen;
2. есть payload example;
3. есть договоренность, что источник временно mock.

Фронт не должен идти вперед, если:

1. shape сущности еще плавает;
2. endpoint может быть переименован;
3. непонятно, какие поля обязательны;
4. mock и real payload отличаются по форме.

---

## Что бэкенду не надо делать

Бэкенду не надо:

1. ждать, пока фронт дорисует экран, чтобы начать следующий endpoint;
2. отгружать “почти готово, но ключи еще могут поменяться” как handoff;
3. считать, что frontend сам “додумает” состояние экрана из сырого payload.

---

## Что считается хорошим handoff для фронтенда

Фронтенд считает handoff хорошим, если есть:

1. endpoint;
2. пример запроса;
3. пример ответа;
4. список ошибок/состояний;
5. пометка:
   - `real`
   - `mock-safe`
   - `temporary demo`.

---

## Что считается хорошим handoff для бэкенда

Бэкенд считает handoff от фронта хорошим, если есть:

1. конкретный экран;
2. конкретная проблема;
3. описание, почему это мешает UI;
4. минимальное изменение, которое решает проблему.

---

## Как не устроить хаос с моками

Моки допустимы только если:

1. UI уже финальный;
2. shape payload совпадает с будущим реальным;
3. mock не размазан по компонентам;
4. позже можно переключить источник данных без редизайна.

Плохой путь:

- жестко зашить данные прямо в экран;
- потом переписывать экран при подключении API.

Хороший путь:

- экран уже работает через composable/store;
- внутри него сегодня mock, завтра API.

---

## Минимальный ежедневный пинг

Каждый пишет:

1. что закончил;
2. что делает;
3. что передает;
4. что блокирует.

Шаблон:

```md
Готово:
- ...

В работе:
- ...

Передаю:
- ...

Блокеры:
- ...
```

## Freeze-пинг перед записью

```md
Снимаем:
- стартовый экран
- голосовой ввод
- вайб-паспорт
- карту рекомендаций
- detail screen

Frozen backend:
- POST /profile/voice
- POST /profile/finalize
- GET /map/locations
- GET /locations/{id}

Frozen frontend:
- порядок экранов
- тексты кнопок
- ключевые CTA

Допустимые моки:
- curated recommendations
- fake 3D preview

Не менять до окончания записи:
- shape payload
- экранный порядок
- основные тексты
```

---

## Критерий нормальной синхронизации

Все идет правильно, если:

1. фронт почти не ждет “когда бэк сделает вообще все”;
2. бэк почти не получает “мы сами выдумали структуру, подстройтесь”;
3. demo и MVP строятся на одних и тех же экранах и сущностях;
4. mock-режимы не ломают будущую архитектуру.
